1. 현재 구축된 이중 정기 백업 시스템 현황 (How Backup Works)
--------------------------------------------------------------------------------
본 시스템은 데이터 유실 리스크를 완벽하게 차단하기 위해 화면 기반 수동 백업(옵션 A)과 
클라우드 서버리스 자동 백업(옵션 B)을 동시에 운영하는 이중 안전장치를 갖추고 있습니다.

■ [옵션 A] 화면 기반 수동 백업 및 일괄 복원 센터 (프론트엔드 통제형)
  - 접속 경로: 시스템 왼쪽 사이드바 메뉴 ➔ [데이터 백업/복원] 클릭
  - 주요 기능:
    ① DB 전체 일괄 백업: [원클릭 DB 전체 백업 파일 다운로드] 클릭 시, 
       인사정보, 급여이력, 연차, 증명서 대장 등 12개 테이블 전체 데이터를 포맷팅하여 
       하나의 통합 데이터 파일(.json)로 내 컴퓨터 [다운로드] 폴더에 즉시 저장합니다.
    ② 개별 테이블 엑셀용 CSV 추출: 데이터베이스 백업 대상 현황 리스트에서 각 테이블의 
       [엑셀용 CSV 다운로드] 버튼을 클릭하면, 엑셀 한글 깨짐 방지 처리(BOM)가 적용된 
       엑셀 호환용 CSV 파일이 개별적으로 내 PC에 다운로드됩니다.
    ③ 데이터베이스 일괄 복원: 예기치 못한 데이터 소실 시, 백업했던 .json 파일을 
       선택하여 올리면 기존 데이터를 안전하게 비운 뒤 외래키 참조 순서에 맞춰 
       100% 정상 시점으로 일괄 복구합니다. (※주의: 기존 데이터 덮어쓰기 주의)
    ④ 정기일 대시보드 알림: 매월 11일이 되면 대시보드 주요 알림 최상단에 
       수동 정기 백업을 실행하도록 가이드하는 권장 경고 배너가 자동으로 활성화됩니다.

■ [옵션 B] GitHub Actions 기반 매월 11일 무인 자동 백업 (백엔드 자동화형)
  - 작동 방식: 매월 11일 한국 시간(KST) 00:00(자정)에 깃허브 Actions 서버가 스케줄러를 
    통해 자동으로 실행되어 Supabase 데이터베이스의 전체 데이터를 안전하게 수집합니다.
  - 저장 위치: 수집된 백업 파일은 프로젝트의 온라인 GitHub 저장소 내 [backups/] 폴더에 
    "MJ_HR_Backup_YYYY-MM-DD.json" 형태로 자동 커밋 및 푸시되어 차곡차곡 안전하게 보관됩니다.
  - ★ 필수 활성화 방법 (Secrets 등록):
    GitHub Actions가 실행될 수 있도록 아래 2가지 보안 비밀 키를 본인의 깃허브 저장소 
    [Settings] ➔ [Secrets and variables] ➔ [Actions] ➔ [New repository secret]에 등록해 주세요.
    * Secret Name: SUPABASE_URL
      ➔ Value 값: https://uwmrwcsawqketzvdtzpy.supabase.co
    * Secret Name: SUPABASE_SERVICE_ROLE_KEY
      ➔ Value 값: Supabase 대시보드 ➔ Settings ⚙️ ➔ API ➔ service_role (secret) 키값 복사해서 입력

2. 향후 보완이 필요한 시스템 보안 취약점 점검 (Remaining Vulnerabilities)
--------------------------------------------------------------------------------
현재 1차 보안 강화를 통해 무단 회원가입과 외부 비인가자의 DB 접근은 철저히 통제되고 있으나, 
기업의 완벽한 규정 준수와 강력한 보안 실현을 위해 보완이 필요한 취약 요인 4가지입니다.

① [🔴 최우선 조치 권장] 주민등록번호 평문 저장에 따른 법적 리스크
  - 현황: 직원의 주민등록번호 13자리가 암호화 없이 Plaintext(평문) 상태로 DB에 저장되고 있습니다.
  - 리스크: 대한민국 [개인정보보호법 제24조의2]에 의거, 주민등록번호는 데이터베이스 저장 시 
    의무적으로 강력한 암호화 알고리즘(AES-256 등)을 적용해 보호해야 합니다. 평문 노출은 불시 
    보안 감사 적발 시 과태료 및 행정처분 등 상당한 법적 리스크를 수반합니다.
  - 개선 방안: React 소스코드 단에서 대칭키 암호화(CryptoJS) 처리 후 암호문을 저장하도록 개선하거나, 
    애초에 마스킹된 데이터(예: 950101-1******)만 수집하여 법적 리스크를 회피하는 설계가 필요합니다.

② [🟡 개선 권장] 회사 직인(인감) 이미지의 Base64 대량 텍스트 저장 방식
  - 현황: 회사 공식 직인 파일이 Base64 텍스트로 변환되어 system_settings 테이블 내 객체에 
    수백 KB 크기로 통합 보관되어 있습니다.
  - 리스크: DB 레코드 크기가 비대해져 전체 데이터 조회 속도를 지연시키고, 민감한 실물 도장 
    이미지가 설정 정보 내에 텍스트 형태로 노출되어 있어 유출/위조 위협에 취약합니다.
  - 개선 방안: Supabase Storage 서비스(비공개 Private 버킷)를 신설하여 직인 이미지를 파일로 
    안전하게 관리하고, 테이블에는 파일 주소만 기록한 뒤 필요할 때만 보안 만료 링크(Signed URL)로 
    불러오도록 분리하는 것이 성능과 보안 측면에서 우수합니다.

③ [🔒 운영 강화] 관리자 로그인 다요소 인증(MFA) 미도입
  - 현황: 현재 시스템은 단순 이메일과 비밀번호로만 로그인합니다.
  - 리스크: 관리자 PC가 악성코드에 감염되거나 비밀번호가 피싱 등으로 유출될 경우, 
    회사 전체의 연봉 정보, 계좌번호, 부양가족 목록 통제권이 단숨에 노출됩니다.
  - 개선 방안: Supabase Auth가 제공하는 표준 TOTP 기반 2단계 인증(MFA)을 적용하여, 로그인 시 
    모바일 OTP 생성 값을 입력하도록 안전장치를 추가하는 것을 검토하십시오.

④ [👥 권한 격리] 사용자 역할 분리 및 역할 기반 접근 제어(RBAC) 미수립
  - 현황: 로그인 세션이 있기만 하면 모든 사용자가 전체 테이블의 등록/수정/삭제 권한을 균등하게 가집니다.
  - 리스크: 향후 일반 직원이나 부서장 계정이 추가될 경우, 다른 사람의 급여를 몰래 보거나 
    임의로 수정할 위험이 있습니다.
  - 개선 방안: 사용자 역할(관리자 / 급여 담당자 / 일반 사원)을 구분하여, 본인의 급여명세서와 
    연차 기록만 볼 수 있도록 RLS 정책을 세분화하여 설계해야 합니다.
